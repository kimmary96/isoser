# Goal

기존 플레이스홀더 랜딩(`frontend/app/page.tsx`)을 대체하지 않고, 별도 경로(`/landing-b`)에 퀴즈 기반 온보딩 랜딩 페이지를 추가한다. 구현 대상은 새 파일 `frontend/app/landing-b/page.tsx`와 여기에 직접 필요한 로컬 스타일링 범위다.

사용자가 3단계 퀴즈(상황 → 수업 방식 → 관심 분야)를 완료하면, 조건에 맞는 프로그램 수와 미리보기를 보여주고, Google 로그인 CTA로 유도한다.

UI 초안 참조 파일은 `cowork/drafts/isoser-landing-B.html`이다.
다만 이 파일은 cowork draft이므로 실행 기준 문서는 본 packet이며, 구현 시에는 draft의 구조와 카피를 참고하되 저장소 실제 경로와 Next.js 구조에 맞게 변환한다.

# User Flow

- 비로그인 사용자가 `/landing-b`에 접근한다
- Hero 영역에서 카피("흩어진 국비 지원 정보, 내 상황에 맞는 것만 골라드립니다")와 소셜 프루프 수치를 본다
- 퀴즈 카드에서 3단계를 순서대로 진행한다
  - Q1: 지금 상황 (취업준비/이직/재취업/창업) — 4개 선택지
  - Q2: 수업 방식 (온라인/오프라인/상관없음) — 3개 선택지
  - Q3: 관심 분야 (AI·데이터/IT·개발/디자인/경영·마케팅 + 직접입력) — 4개 선택지 + 텍스트 입력
- 각 단계에서 선택지를 고르면 "다음" 버튼이 활성화된다
- Q3 완료 시 결과 화면으로 전환된다
- 결과 화면: 필터링된 프로그램 수(앵커링 숫자) + 총 4개의 프로그램 카드 + 1~2번째 정상 노출 + 3~4번째 blur 처리
- Google 로그인 CTA 버튼 클릭 시 `/login`으로 이동
- 하단: 마감 임박 칩, 기능 소개 4개 카드, "다시 찾아볼까요?" 리셋 버튼
- 리셋 시 퀴즈 화면으로 돌아간다

# UI Requirements

- Pretendard 폰트 사용 (기존 프로젝트와 동일)
- 색상 시스템: mockup의 CSS 변수 체계 그대로 사용 (--blue: #2563EB, --fire: #F97316 등)
- Hero: 도트 패턴 배경 + 블러 블롭 장식 + fadeUp 애니메이션
- 퀴즈 카드: 진행바(1/3 → 2/3 → 3/3), 선택지 그리드(2열 기본, Q2는 3열), 선택 시 하이라이트
- Q3 직접입력: 텍스트 필드 포함, 기존 선택지와 배타적 선택
- 결과 화면: 큰 숫자 앵커링 + 태그 칩 + 미리보기 카드(blur 오버레이) + CTA 박스
- 하단: 마감 임박 행(빨간 칩), 기능 그리드(4열), 리셋 섹션
- 반응형: 모바일에서 퀴즈 선택지 1열, 기능 그리드 2열
- 로딩 상태: 별도 데이터 fetch가 없으므로 불필요 (정적 데이터 기반)

# Acceptance Criteria

1. `/landing-b` 경로에서 퀴즈 기반 랜딩 페이지가 정상 렌더링된다
2. 3단계 퀴즈가 순서대로 진행되고, 선택 없이 "다음"을 누를 수 없다
3. Q3 직접입력과 선택지가 배타적으로 동작한다
4. 결과 화면에서 조건 태그가 정확히 표시된다
5. 결과 화면에는 총 4개의 프로그램 카드가 렌더링되고, 1~2번째는 정상 노출, 3~4번째는 blur overlay 처리된다
6. CTA 클릭 시 `/login`으로 이동한다
7. "다시 찾아볼까요?" 클릭 시 퀴즈 화면으로 초기화된다
8. 기존 `frontend/app/page.tsx`는 변경하지 않는다
9. 모바일(375px)과 데스크톱(1280px) 모두에서 레이아웃이 깨지지 않는다

# Constraints

- 기존 `frontend/app/page.tsx`를 건드리지 않는다. 별도 경로 추가만 허용한다
- 퀴즈 결과 데이터는 정적 하드코딩이다. 실제 API 연동은 이 Task 범위 밖이다
- 프로그램 미리보기 리스트도 정적 더미 데이터 사용
- 소셜 프루프 수치(847개, 134곳)도 정적 값
- 기존 프로젝트의 Tailwind 설정과 충돌하지 않도록 주의
- `use client` 지시어 필요 (상태 관리가 있으므로)
- 새 경로 구현에 필요한 최소 범위에서만 파일을 추가한다. 공용 스타일/레이아웃 전역 수정은 피한다

# Non-goals

- 실제 프로그램 DB 연동 (정적 데이터만)
- 퀴즈 응답 저장 또는 분석
- A/B 테스트 인프라 (라우팅 분기 로직)
- 기존 랜딩 페이지 교체 판단
- SEO 메타데이터 최적화

# Edge Cases

- 사용자가 Q3에서 직접입력 중 기존 선택지를 다시 클릭하면, 입력 필드가 초기화되어야 한다
- 직접입력 필드에 빈 문자열만 입력한 경우 "다음" 버튼이 비활성 상태를 유지해야 한다
- 결과 화면에서 브라우저 뒤로가기 시 퀴즈 화면으로 돌아가는 동작은 이 Task에서 보장하지 않아도 된다
- 리셋 후 스크롤이 최상단으로 이동해야 한다
