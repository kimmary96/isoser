- 고용24 상세 페이지 HTML fallback 파싱은 `backend/rag/collector/work24_detail_parser.py`에 분리되어 있으며, `scripts/program_backfill.py`는 이 파서를 호출해 기존 빈 row의 기관, 주소, 기간, 훈련비, 지원금, 만족도, 연락처 메타를 보강한다. 상세 HTML에 수강신청/모집/접수 마감일이 명시된 경우에만 `deadline`과 `compare_meta.application_deadline`으로 보강하고, 훈련 종료일을 모집 마감일로 추정하지 않는다. 백엔드 상세 API는 고용24 상세 화면의 신청 종료일도 같은 resolved deadline을 사용해 `close_date`가 비어 있어도 `compare_meta.application_deadline` 또는 신뢰 가능한 `deadline`이 있으면 표시한다. 상세 HTML의 `훈련유형`, `주야구분`, `주말여부`, `훈련시간`은 `compare_meta`에 보존해 목록 참여시간/지원유형 표시의 원천으로 쓴다.
- scheduler의 Supabase upsert는 `source_unique_key`가 있으면 이를 우선 conflict target으로 사용하고, collector 결과 dedupe도 `source_unique_key`를 `title/source`보다 먼저 사용해 같은 제목의 고용24 다른 회차를 합치지 않는다. `source_unique_key`가 없는 source row는 normalizer가 `source + URL + title` 해시 기반 `urltitle:*` key를 생성해 legacy `(title, source)` unique 제거 후에도 반복 sync 중복을 줄인다. `backend/routers/admin.py`의 row-by-row fallback도 `source_unique_key`가 있는 row는 legacy `hrd_id` 또는 `(title, source)` row로 병합하지 않도록 방어한다. `20260422190000_add_programs_source_unique_key.sql`과 `20260422201000_fix_programs_source_unique_key_conflict_index.sql`은 `programs.source_unique_key`와 Supabase REST upsert가 사용할 수 있는 unique index를 추가하고, `20260423143000_relax_programs_legacy_unique_constraints.sql`은 고용24의 `trprId + trprDegr + trainstCstId` 단위 회차 저장을 막는 legacy `programs_unique(title, source)` 및 `hrd_id` unique 제약 제거 migration이다. `20260422203000_add_programs_search_text_index.sql`은 검색 후보 축소용 `search_text` generated column과 trigram index를 추가했고, `20260423112000_refine_programs_search_metadata.sql`은 이 검색 텍스트와 category detail 추론에서 `compare_meta` 전체 JSON 대신 교육 관련 allowlist key만 사용하도록 보정한다. `20260422213000_add_programs_cost_time_filters.sql`은 `programs.cost_type`, `programs.participation_time` 컬럼과 분류 trigger/index를 추가해 비용/참여 시간 필터의 DB 저장 기준을 마련한다.
- `HRDCollector`는 optional source로 전환되어 기본값 `ENABLE_HRD_COLLECTOR=false` 상태에서는 scheduler가 `skipped_disabled`로 기록하고 실행하지 않는다.
- `ENABLE_HRD_COLLECTOR=true`여도 `HRD_API_KEY` 또는 `HRDNET_API_KEY`가 없으면 scheduler는 실패 대신 `skipped_missing_config`로 기록한다.

## End-to-end packet flow
1. planner가 `cowork/packets/<task-id>.md`에 task packet 원본을 만든다.
2. `cowork_watcher.py`가 현재 저장소와 packet frontmatter를 검사해 `cowork/reviews/<task-id>-review.md`를 만든다.
3. reviewer는 review 문서를 보고 `cowork/packets/<task-id>.md` 원본을 수정하거나 보강한다.
4. packet이 바뀌면 cowork watcher는 최신 packet 기준으로 review를 다시 생성하거나 stale review를 막는다.
5. 승인되면 cowork watcher는 `cowork/packets/<task-id>.md` 최신본을 `tasks/inbox/` 또는 `tasks/remote/`로 복사한다.
6. local path라면 `watcher.py`가 `tasks/inbox/`에서 packet을 집어 `tasks/running/`으로 옮기고 Codex를 실행한다.
7. local watcher supervisor는 inspector handoff를 먼저 만들고, 이어서 implementer가 구현과 result report를 만든다. 일반 코드 task는 verifier가 최종 검증 artifact를 남기고, docs task는 watcher가 경량 verification artifact를 대신 남긴다.
8. watcher는 실행 결과에 따라 packet을 `tasks/done/`, `tasks/drifted/`, `tasks/blocked/`, `tasks/review-required/`로 이동한다.
9. `tasks/drifted/`와 `tasks/blocked/`는 자동 복구가 가능하면 `tasks/inbox/`로 재큐잉되고, 아니면 다시 `cowork/packets/` review 흐름으로 에스컬레이션된다. `tasks/review-required/`는 verifier가 수동 검토를 요청한 전용 큐이며, 검토가 끝난 packet은 그 큐에 남기지 않고 `tasks/archive/` 또는 후속 execution queue로 정리한다.

## Folder semantics
- `cowork/packets/`: 사람이 계속 수정하는 원본 packet
- `cowork/reviews/`: 원본 packet에 대한 review 이력
- `tasks/inbox/`: 승인된 최신 packet 사본의 실행 대기열
- `tasks/running/`: 현재 실행 중인 packet
- `tasks/done/`: 성공적으로 끝난 execution packet
- `tasks/blocked/`: 메타데이터 누락, 실패, 외부 의존성 등으로 멈춘 execution packet
- `tasks/drifted/`: 계획 기준과 현재 코드가 어긋나 재검토가 필요한 execution packet
- `tasks/review-required/`: verifier가 사람 검토 후 재승인을 요구한 살아 있는 execution packet만 두는 큐
- `tasks/archive/`: 중복 packet, review 처리 완료 packet, 재실행 대상이 아닌 stale packet을 보관하는 아카이브

## Current behavior notes
- PDF 이력서 파서는 Gemini 호출 실패/쿼터 초과 시 원문 기반 fallback 파싱을 수행하며, `경력` 섹션은 `프로젝트` 헤더에서 종료해 프로젝트가 프로필 경력으로 섞이지 않도록 처리한다.
- 활동 역할/팀 구성 파싱은 `백엔드 개발자 (5인: ...)`와 `백엔드 개발자 (5인 팀: ...)` 표기를 모두 지원하며, scorer가 소개형/성과형/무시 문장을 점수 기반으로 분류한다. PDF parser rule table과 scorer weight는 `backend/chains/pdf_parser_rules.py`에 모아 두고, 역할명 단독 줄은 기여내용에서 제외하고 구현/개발/성과 문장은 기여내용으로 분류한다. scorer classification은 `metric_signal`, `keyword_signal`, `role_only` 같은 세분화된 reason을 반환하며, `ISOSER_PDF_PARSE_DEBUG_SCORER=1`일 때 실제 파싱 중 debug log로 남긴다. PDF 원문 회귀 케이스는 `backend/tests/fixtures/pdf_texts/`, expected snapshot은 `backend/tests/fixtures/pdf_expected/`, 실제 PDF e2e fixture는 `backend/tests/fixtures/pdf_files/`로 관리한다.
- 추천 프로그램 API는 `relevance_score`와 `urgency_score`를 분리해서 반환하며, 카드 UI는 관련도 배지에 `relevance_score`를 사용하고 마감 7일 이내만 별도 마감 칩으로 표시한다.
- `backend/routers/programs.py`는 기존 `POST /programs/recommend`를 유지한 채 `GET /programs/recommend/calendar`를 추가해 `{ items: [...] }` 계약으로 캘린더 전용 추천을 제공하고, 이 경로에서만 만료 프로그램 제외 + `final_score desc`, `deadline asc` 정렬을 적용한다. 캘린더 추천 item은 기존 맞춤 추천 카드에서 쓰던 `fit_keywords`, `relevance_reasons`, `score_breakdown`, `relevance_grade`, `relevance_badge`도 함께 반환해 대시보드에서 관련도 배지와 추천 근거를 표시할 수 있다.
- recommendation cache read는 저장된 `final_score`를 그대로 신뢰하지 않고 `relevance_score * 0.6 + urgency_score * 0.4`로 재계산하며, component score가 하나라도 없으면 stale cache로 보고 fresh recommendation path로 우회한다. 최신 캐시 컬럼(`similarity_score`, `relevance_score`, `urgency_score`, `final_score`, `generated_at`)이 아직 운영 DB에 없으면 legacy `score`/`created_at` 캐시를 읽는 fallback을 사용하고, `20260423123000_align_recommendations_cache_schema.sql` migration은 운영 `recommendations` 테이블을 현재 캐시 계약에 맞춘다.
- `frontend/app/api/dashboard/recommend-calendar/route.ts`와 `frontend/lib/api/app.ts`는 새 캘린더 추천 BFF/helper를 제공하며, 비로그인 사용자도 `relevance_score = 0`을 유지한 `{ items: CalendarRecommendItem[] }` 응답을 받을 수 있다. 백엔드 캘린더 추천은 3.5초 timeout 후 모집 마감순 공개 프로그램 fallback으로 전환하고, fallback 목록 fetch도 2.5초 timeout 후 Supabase direct fallback으로 넘어가 발표 퍼널 보호와 초기 화면 대기 시간을 줄인다. Supabase direct fallback은 고용24 `deadline=end_date` 오염 row를 제외하고, backend calendar item도 resolved deadline이 없는 프로그램은 날짜형 캘린더 추천에서 제외한다.

- `frontend/app/dashboard/portfolio/page.tsx`는 세션스토리지에 남은 포트폴리오 변환 결과가 있으면 기존 미리보기를 보여주고, 없으면 성과 저장소 활동 목록에서 활동을 선택해 `/activities/convert` 기반 포트폴리오 초안을 직접 생성할 수 있다. 생성 결과는 `portfolios.portfolio_payload`에 저장되어 재진입 시 저장된 초안을 다시 열 수 있으며, 미리보기 화면에서 브라우저 인쇄 기반 `PDF로 저장`을 제공한다.
- 비교 페이지는 `POST /programs/batch`로 기본 프로그램을 한 번에 조회하고, `POST /programs/details/batch` 상세 응답을 함께 붙여 기본 정보, 일정, 비용·지원, 대상·모집, 소개 섹션을 비교한다. 상세 batch 호출이 실패해도 기존 목록 데이터 fallback으로 화면은 유지된다. 비교 프로그램 선택 모달의 찜한 프로그램 탭은 열릴 때마다 현재 Supabase 로그인 사용자 기준 `program_bookmarks`를 재조회해 최신 북마크를 보여준다. 전체 검색 탭은 별도 필터 없이 검색어 기반 프로그램 검색만 제공한다.

- 고용24/K-Startup/SeSAC 프로그램 매핑은 제목, 설명, 대상, 사업 유형, 고용24 NCS 코드에서 보수적인 스킬 키워드를 추출해 `programs.skills` 배열로 넘기고, normalizer는 이를 중복 제거된 text list로 저장 payload에 포함한다.
- compare relevance 응답은 기존 점수 필드 외에 `region_match_score`, `matched_regions`, `relevance_reasons`, `score_breakdown`, `relevance_grade`, `relevance_badge`, `fit_label`, `fit_summary`, `readiness_label`, `gap_tags`를 함께 반환한다. 프로필 지역 정보가 없으면 기존 관련도 점수를 유지하고, 지역 정보가 있으면 프로그램 지역/위치 텍스트와의 보수적 일치 신호를 최대 15% 반영한다. 지역 매칭은 국내 시/도 정규화, 수도권/충청권 등 1차 인접권역, 온라인/혼합형 키워드를 구분한다.
- 프로그램 목록 페이지는 상단 필터, 1줄 가로 스크롤 마감 임박 레일, 엑셀형 전체 프로그램 테이블로 구성된다. 맞춤 추천 섹션은 목록 화면에서 제거되어 전체 프로그램 비교 탐색을 우선한다. `/programs` 본문 컨테이너는 `max-w-[1680px]` 기준으로 Program Search, Closing Soon, 전체 프로그램 카드가 같은 좌우 폭에 정렬된다. Closing Soon은 전역 모집중 후보를 기준으로 유지되며, 우선 D-7 이내 프로그램을 보여주고 그런 row가 없으면 현재 공개된 프로그램 중 더 빨리 확인해야 할 순서대로 fallback 카드 6개를 유지한다. 이 레일은 일반 `ProgramCard` 대신 마감일, 제목, 핵심 chip, 2줄 설명만 표시하는 압축 카드로 렌더링해 필터와 전체 목록 사이의 세로 간격을 줄인다. 전체 목록 테이블은 제목/source가 있는 프로그램을 노출하며, 모집 마감일이 resolved 되지 않은 row는 `마감일 미확인`으로 표시한다. 제목 링크는 상세 이동, 별 버튼은 dashboard bookmark BFF를 경유해 찜 mutation을 수행한다. 테이블의 프로그램 과정은 backend `display_categories`를 우선 사용해 대표 카테고리 1~2개를 badge로 표시하고, 참여 시간은 `participation_mode_label`과 `participation_time_text`를 함께 보여주되 Work24 총 시간이 있으면 `48시간`처럼 축약해 노출한다. 선발절차·키워드 칸은 `selection_process_label`과 `extracted_keywords` chip을 표시하고, 키워드 성격에 따라 색상을 달리한다. 기본 전체 화면에서는 promoted program 3건까지를 별도 섹션이 아니라 메인 리스트 상단에 병합해 보여주고, 검색이나 카테고리/필터가 걸리면 이 상단 고정을 해제한다. 고용24처럼 `category_detail`이 비어 있는 row도 세부 카테고리 필터에서 큰 카테고리 후보와 파생 카테고리/alias 후처리로 매칭하며, `근로자원격훈련` 같은 원천 target은 목록 응답의 `teaching_method=온라인` 표시로 보정한다. 전체 프로그램 테이블에서는 채용연계 열을 표시하지 않고, 해당 원본 데이터는 비교/상세 등 다른 화면에서 재사용할 수 있도록 API 필드에는 유지한다. 테이블은 내부/카드 가로 스크롤 없이 전체 카드 폭 안에서 고정 레이아웃으로 펼쳐지고, 긴 제목과 키워드는 줄바꿈으로 수용한다. 파생 키워드가 없으면 chip 블록만 숨기고, 선발 절차가 없으면 중복 dash 없이 `선발 절차 없음`만 표시한다. 로그인 사용자는 서버 렌더링 시 `program_bookmarks`에서 기존 찜 program id를 읽어 초기 별 상태에 반영하며, `/programs` 화면 안에서는 같은 program id 카드/행이 공유 bookmark state를 사용해 찜 변경을 즉시 동기화한다. 필터는 지역, 비용, 참여 시간, 운영 기관, 추천 대상 조건을 URL query와 backend 목록/count API에 연결한다. 다중 필터는 항목 클릭 후 메뉴가 자동으로 닫히고, 정렬은 기본 정렬, 인기순, 마감 임박순, 개강 빠른순, 비용 낮은순, 비용 높은순, 짧은 기간순, 긴 기간순 선택 즉시 적용된다. 운영 기관과 추천 대상 옵션은 `GET /programs/filter-options`가 현재 검색/카테고리/지역/수업방식/모집 상태 조건에 맞는 실제 프로그램 데이터에서 추출하며, 옵션 조회 실패나 빈 결과에서는 기존 정적 옵션으로 fallback한다.
