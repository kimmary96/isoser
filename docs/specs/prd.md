# 이소서 (Isoser) PRD

> 국가 취업 지원 정보 허브 + AI 코치 기반 지원 문서 작성 도구
> Product Requirements Document
> v2.3 | 2026-04-20

## 1. 제품 요약

### 1.1 한 줄 정의

이소서는 파편화된 공공 취업 지원 프로그램 정보를 한곳에 모으고, 사용자의 활동 데이터를 기반으로 이력서·자기소개서·지원 문서를 빠르게 만들 수 있게 돕는 서비스다.

### 1.2 제품 정체성

- 단순 AI 글쓰기 서비스가 아니다.
- 정부·공공 취업 지원 정보 탐색과 지원 문서 작성을 하나의 흐름으로 묶는 제품이다.
- 핵심 경쟁 대상은 ChatGPT 단독 사용이 아니라 고용24, HRD넷, HRD클럽처럼 정보는 있으나 개인화와 문서화가 약한 서비스다.

### 1.3 현재 제품 단계

- `v2` 방향은 명확하다.
- 다만 현재 구현은 완성형 정보 허브가 아니라, `문서 작성 도구 v1` 위에 `프로그램 허브 기능 v2`를 붙이는 전환 단계다.
- 따라서 이 문서는 목표 PRD이면서 동시에 2026-04-14 기준 구현 현황을 함께 기록한다.

## 2. 문제 정의

### 2.1 사용자 문제

1. 취업 지원 프로그램 정보가 여러 사이트에 흩어져 있어 찾기 어렵다.
2. 마감일, 대상, 지원 조건을 직접 추적해야 해서 탐색 비용이 크다.
3. 지원을 결심한 뒤에도 이력서와 자기소개서를 매번 다시 조립해야 한다.
4. 활동, 프로젝트, 경력 성과가 흩어져 있어 재사용이 어렵다.
5. AI를 써도 최신 공공 프로그램 정보와 내 활동 데이터가 연결되지 않는다.

### 2.2 우리가 푸는 방식

- 프로그램 탐색: 공공 취업 지원 프로그램을 수집하고 목록화한다.
- 체류 도구: 활동 저장소, AI 코치, 이력서, 자기소개서 기능을 무료 도구로 제공한다.
- Lock-in: 사용자가 정리한 활동과 문서 자산을 누적 저장한다.
- 전환: 정보 탐색이 문서 작성으로 이어지게 만든다.

## 3. 타깃 사용자

### 3.1 1차 타깃

- 부트캠프 수강생
- 공백기 취업 준비생
- 직무 전환 준비자

### 3.2 2차 타깃

- 경력 1년차 이상 이직 준비자
- 재취업 준비자
- 활동 정리를 도와주는 코치·멘토

### 3.3 핵심 사용 맥락

- 내게 맞는 프로그램을 찾고 싶다.
- 지금 가진 활동으로 어떤 문서를 만들 수 있는지 알고 싶다.
- 활동 설명을 STAR 구조와 직무 맞춤 표현으로 보완하고 싶다.
- 여러 지원처에 맞게 이력서와 자기소개서를 재조합하고 싶다.

## 4. 제품 목표

### 4.1 비즈니스 목표

- 취업 지원 정보 허브로 지속 유입을 만든다.
- 문서 작성 도구로 체류 시간을 늘린다.
- 활동 데이터 누적으로 재방문과 잔존을 높인다.
- 장기적으로 광고, 스폰서드 프로그램, 프리미엄 문서 기능으로 확장한다.

### 4.2 제품 목표

- 정보 탐색에서 문서 작성까지 한 번에 이어지는 사용자 흐름을 제공한다.
- 사용자가 매번 새 문서를 쓰는 대신 저장된 활동을 재사용하게 만든다.
- AI는 대필보다 피드백과 제안 중심으로 동작하게 한다.

### 4.3 성공 지표

- 프로그램 상세 조회 대비 북마크 전환율
- 활동 1개 이상 저장한 사용자 비율
- AI 코치 사용 후 활동 저장 완료율
- 이력서 생성 완료율
- 자기소개서 재편집/재사용률
- 7일 재방문율

- 템플릿 선택 UI에 비해 PDF 출력 결과는 아직 단일 기본형 중심이다.

### 6.2 현재 제약과 갭

1. 프로그램 허브는 `고용24`, `K-Startup`, 일부 HTML 수집 소스 중심이며 사업계획서가 상정한 전체 공공지원 허브 범위에는 아직 못 미친다.
2. 추천 결과는 API와 대시보드 경로가 연결돼 있지만, 북마크와 문서 작성으로 이어지는 완성형 퍼널은 더 다듬어야 한다.
3. 포트폴리오는 변환 결과 미리보기 수준이라 저장, 배포, 템플릿화가 약하다.
4. PDF 출력은 기본 이력서 중심이며 템플릿 다양성이 부족하다.
5. 일부 AI 기능은 별도 Gemini 키와 백엔드 설정에 의존한다.
6. 광고 상품, 스폰서드 슬롯, 자연어 정보 검색 챗봇은 사업계획 레벨이며 운영 기능은 아직 아니다.

## 7. 핵심 사용자 플로우

### 7.1 문서 작성 플로우

1. 사용자가 공개 허브를 둘러보거나 로그인으로 진입한다.
2. 이력서 PDF를 업로드해 프로필과 활동 초안을 만든다.
3. 활동 저장소에서 활동을 정리하고 STAR 구조를 보완한다.
4. AI 코치로 피드백과 rewrite suggestion을 받는다.
5. 활동을 선택해 이력서를 생성하고 PDF로 다운로드한다.

### 7.2 공고 대응 플로우

1. 사용자가 공고 텍스트 또는 공고 이미지/PDF를 입력한다.
2. 시스템이 활동과 공고를 비교해 매칭 점수와 갭을 분석한다.
3. 사용자는 추천 활동과 rewrite suggestion을 참고해 문서를 수정한다.
4. 수정한 내용을 이력서 또는 자기소개서에 반영한다.

### 7.3 프로그램 허브 플로우

1. 사용자가 프로그램 목록 또는 인기 프로그램을 조회한다.
2. 상세 정보를 확인하고 북마크한다.
3. 장기 목표는 여기서 개인화 추천과 문서 작성 진입이 이어지는 것이다.

## 8. 기능 요구사항

### F1. PDF 파싱 온보딩

- 입력: 이력서 PDF
- 처리: 텍스트 추출 후 LLM 구조화
- 결과: 프로필 초안과 활동 초안 반환

### F2. 프로필 관리

- 입력: 이름, 연락처, 소개, 링크, 이미지, 경력/학력/기술 정보
- 처리: `profiles` upsert 및 Storage 업로드
- 결과: 대시보드 프로필 상태 갱신

### F3. 활동 저장소

- 입력: 활동 메타데이터, 본문 설명, STAR 필드
- 처리: `activities` CRUD
- 결과: 활동 카드, 상세 화면, 문서 조합 소스

### F4. 활동 구조 변환

- 입력: 활동 1건
- 처리: 규칙 기반 STAR 변환 또는 포트폴리오 구조 변환
- 결과: STAR 초안 또는 포트폴리오 섹션 초안

### F5. AI 코치

- 입력: 활동 설명, 목표 직무, 섹션 타입, 대화 이력
- 처리: 구조 진단, 누락 요소 탐지, rewrite suggestion 생성, 세션 저장
- 결과: 피드백, suggestion, iteration count, 복원 가능한 세션

### F6. 공고 분석

- 입력: 공고 텍스트, 사용자 활동, 프로필 문맥
- 처리: 공고-활동 매칭 점수 계산, 강점/갭 분석
- 결과: 총점, 세부 점수, 추천 활동, 지원 추천도

### F7. 공고 추출

- 입력: 공고 이미지 또는 PDF
- 처리: OCR 또는 PDF 텍스트 추출
- 결과: 분석 가능한 공고 본문

### F8. 공고 기반 리라이팅

- 입력: 사용자 ID, 공고 본문, 목표 직무, 활동 ID, 섹션 타입
- 처리: 저장된 활동과 공고 요구사항을 함께 참조해 rewrite suggestion 생성
- 결과: 활동별 suggestion 1~3개

### F9. 기업 정보 요약

- 입력: 회사명
- 처리: 외부 리서치 체인 호출
- 결과: 지원 판단에 필요한 요약 정보

### F10. 이력서 생성과 저장

- 입력: 제목, 타깃 직무, 선택 활동 목록
- 처리: `resumes` 저장, 선택 활동 참조
- 결과: 문서 저장소 반영, PDF 다운로드 가능 상태

### F11. 자기소개서 저장소

- 입력: 회사, 직무, 문항, 답변, QA 배열
- 처리: `cover_letters` CRUD
- 결과: 재사용 가능한 자기소개서 자산 저장

### F12. 프로그램 허브

- 입력: 카테고리, 지역, 범위, 관리자 수집 파라미터
- 처리: 프로그램 목록 조회, 상세 조회, 북마크, 관리자 동기화
- 결과: 프로그램 탐색 및 저장

## 11. 비기능 요구사항

### 11.1 보안

- 사용자 데이터는 본인 범위로 제한되어야 한다.
- service role key는 서버에서만 사용한다.
- OAuth 세션과 API 토큰 검증이 일관되어야 한다.

### 11.2 성능

- 대시보드 주요 뷰는 첫 로딩 체감이 과도하게 느리지 않아야 한다.
- 공고 분석과 AI 코치 응답은 실패 시 사용자에게 명확한 오류를 보여줘야 한다.
- 프로그램 동기화는 배치 단위로 안정적으로 동작해야 한다.

### 11.3 운영

- 프론트엔드: Next.js App Router
- 백엔드: FastAPI
- DB/Auth/Storage: Supabase
- 추천 및 코치 보조 인덱스: Chroma
- 런타임 제약: 백엔드는 Python 3.10 계열 기준

### 11.4 문서 품질 원칙

- AI는 최종본 대필보다 피드백과 선택 가능한 초안 제안에 집중한다.
- 사용자는 저장된 자산을 재조합해 문서를 만든다.
- 프로그램 허브는 단순 목록이 아니라 문서 작성 행동으로 이어져야 한다.

## 12. 우선순위

### P0

1. 프로그램 추천 결과와 북마크/문서 작성 플로우 연결 강화
2. 이력서 PDF 품질 개선과 템플릿 구조화
3. 프로그램 허브 데이터 소스 확대와 품질 안정화

### P1

1. 포트폴리오 전용 편집 화면과 출력
2. 자기소개서와 공고 분석 자동 연결
3. 문서 저장소 검색/필터/정렬 강화

### P2

1. 광고 슬롯/스폰서드 프로그램 운영 구조
2. 다중 공공 데이터 소스 확장
3. 프리미엄 문서 기능 실험

## 13. 리스크와 의존성

- 공공 프로그램 데이터 품질과 수집 안정성에 의존한다.
- Gemini 및 외부 AI 호출 실패 시 fallback 품질이 중요하다.
- 프론트와 백엔드의 기능 연결 상태가 문서보다 뒤처질 수 있다.
- 프로그램 추천은 데이터 확보량과 인덱스 품질에 큰 영향을 받는다.
