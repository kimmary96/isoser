# P0 Release Test Plan

이 문서는 링크 공유 직전에 확인할 P0 기능 테스트와 테스트 계정 운영 방식을 정리한다.

## 1. 테스트 원칙

- 개인 Google 계정 비밀번호는 공유하지 않는다.
- 테스트는 가능하면 별도 Google 테스트 계정으로 수행한다.
- 실제 사용자 데이터와 섞이지 않도록 테스트 계정 이름, 이메일, 활동명에 `TEST` 접두어를 붙인다.
- 실패하면 바로 다음 항목으로 넘어가지 말고, 실패 화면, URL, 시간, 입력값을 기록한다.
- 설정 검증이 아니라 실제 사용자가 클릭하는 기능을 끝까지 확인한다.

## 2. 테스트 계정 운영 방식

### 권장 방식

1. 테스트 전용 Google 계정을 만든다.
2. Supabase Auth에서 해당 계정으로 로그인 가능한지 확인한다.
3. 테스트 중 생성한 프로필, 활동, 문서, 찜 데이터는 모두 테스트 계정 소유로 남긴다.
4. 테스트 종료 후 필요한 경우 Supabase에서 테스트 계정 데이터를 삭제하거나 그대로 회귀 테스트용 seed처럼 유지한다.

### Codex가 같이 확인해야 하는 경우

- 실계정 비밀번호를 공유하지 않는다.
- 사용자가 브라우저에서 직접 로그인한 뒤, 로그인된 세션 상태에서 화면/결과를 알려주거나 같이 점검한다.
- 자동화가 꼭 필요하면 일회용 테스트 계정을 만들고, 테스트 직후 비밀번호를 변경하거나 계정을 삭제한다.
- 2FA, 복구 이메일, 개인 계정 비밀번호는 공유하지 않는다.

## 3. P0 통과 기준

아래가 모두 통과하면 링크 공유 가능 상태로 본다.

- `/api/health/config`가 `status: ok`를 반환한다.
- 공개 랜딩과 프로그램 탐색이 열린다.
- 프로그램 목록이 비어 있지 않다.
- Google 로그인과 로그아웃이 된다.
- 신규 또는 테스트 계정으로 대시보드에 진입한다.
- 프로필 저장이 된다.
- 활동 생성, 수정, 삭제가 된다.
- 프로그램 찜이 대시보드에 반영된다.
- AI 기능 1개 이상이 성공한다.
- 이력서 또는 포트폴리오 문서 저장과 PDF 다운로드가 된다.
- 모바일 화면에서 핵심 CTA가 눌린다.
- 다른 계정의 데이터가 보이지 않는다.

## 4. 공개 페이지 테스트

### 4.1 랜딩

- `https://isoser.vercel.app` 접속
- `/landing-c`로 이동하거나 랜딩 화면이 정상 표시되는지 확인
- 헤더의 `프로그램 탐색`, `비교`, `대시보드`, `로그인` 클릭
- Live Board와 Opportunity feed에 프로그램 카드가 표시되는지 확인
- 모바일에서 첫 화면 CTA가 접히거나 겹치지 않는지 확인

### 4.2 프로그램 목록

- `/programs` 접속
- 전체 프로그램 수와 목록이 표시되는지 확인
- 검색어 `AI`, `개발`, `창업`으로 검색
- 지역, 비용, 참여 시간, 운영 기관 필터를 각각 1개 이상 적용
- 정렬을 마감 임박순, 인기순, 비용 낮은순으로 바꿔본다.
- 페이지네이션을 이동한다.
- 빈 결과 검색어를 넣었을 때 빈 상태 안내가 정상인지 확인한다.

### 4.3 프로그램 상세

- 프로그램 목록에서 제목을 클릭한다.
- 상세 페이지에서 일정, 비용, 대상, 설명, 원문 링크가 표시되는지 확인한다.
- 원문/신청 링크가 새 탭 또는 원래 의도한 방식으로 열린다.
- 상세 페이지 새로고침 후에도 500 오류가 나지 않는지 확인한다.

### 4.4 비교

- `/compare` 접속
- 프로그램 2개 이상 추가
- 프로그램 삭제와 교체
- 비교 표의 일정, 비용, 대상, 소개 섹션 확인
- 데이터가 없는 항목은 `정보 없음` 또는 `데이터 미수집`으로 표시되는지 확인

## 5. 인증 테스트

- 비로그인 상태에서 `/dashboard` 접근 시 `/login?redirectedFrom=...`로 이동하는지 확인
- Google 로그인 버튼 클릭
- 로그인 후 원래 가려던 경로 또는 대시보드로 돌아오는지 확인
- `/login`에 다시 접근했을 때 로그인된 사용자는 적절한 내부 경로로 이동하는지 확인
- 로그아웃 후 `/dashboard` 접근이 다시 차단되는지 확인

## 6. 대시보드 P0 테스트

### 6.1 프로필

- `/dashboard/profile` 접속
- 이름, 전화번호, 지역, 포트폴리오 링크 중 1개 이상 수정 후 저장
- 새로고침 후 값이 유지되는지 확인
- 프로필 이미지 업로드가 필요한 경우 정상 이미지 1개 업로드

### 6.2 활동 저장소

- `/dashboard/activities` 접속
- 새 활동 생성
- 활동명: `TEST 런칭 점검 활동`
- 활동 유형, 조직, 기간, 역할, 기여내용 입력
- 저장 후 목록에 표시되는지 확인
- 상세 페이지에서 수정 후 저장
- 이미지 업로드가 필요한 경우 정상 이미지 1개 업로드
- 삭제 또는 테스트 데이터 유지 여부 결정

### 6.3 프로그램 찜과 캘린더

- `/programs`에서 프로그램 1개 찜
- `/dashboard`에서 찜한 훈련 섹션에 반영되는지 확인
- 추천 카드가 표시되는지 확인
- 캘린더 적용 버튼을 눌러 대시보드 캘린더에 표시되는지 확인

## 7. AI 기능 P0 테스트

비용과 지연을 줄이기 위해 아래 중 2개만 우선 확인한다.

### 7.1 매칭 분석

- `/dashboard/match` 접속
- 회사명: `TEST 회사`
- 직무: `프론트엔드 개발자`
- 공고 본문에 짧은 테스트 공고 입력
- 분석 실행
- 점수, 총평, 상세 결과가 표시되는지 확인
- 저장 목록에 결과가 남는지 확인

### 7.2 자기소개서 또는 활동 코치

- 자기소개서 또는 활동 상세 코치 화면에서 짧은 질문 입력
- AI 응답이 표시되는지 확인
- 실패 시 사용자에게 에러 메시지가 보이는지 확인

### 7.3 이력서 공고핏 문장

- `/dashboard/resume` 접속
- 테스트 활동 선택
- 지원 직무와 짧은 공고 본문 입력
- 문장 후보 생성
- 후보 적용, 해제, 직접 편집 확인

## 8. 문서/PDF P0 테스트

### 8.1 이력서

- `/dashboard/resume` 접속
- 프로필과 활동이 불러와지는지 확인
- 활동 1개 이상 선택
- 문서 생성 또는 저장 실행
- `/dashboard/documents`에서 저장본 확인
- PDF 다운로드가 시작되는지 확인

### 8.2 포트폴리오

- `/dashboard/portfolio` 접속
- 활동 1개 이상 선택
- 초안 생성
- 프로젝트 제목/요약을 수정
- 저장
- 문서 저장소에서 preview가 보이는지 확인
- PDF 다운로드가 시작되는지 확인

## 9. 모바일 테스트

- 모바일 폭에서 `/landing-c`, `/programs`, `/login`, `/dashboard` 확인
- 주요 CTA가 화면 밖으로 밀리지 않는지 확인
- 프로그램 테이블 또는 카드가 읽을 수 있는지 확인
- 모달 footer 버튼이 화면 아래에 가려지지 않는지 확인

## 10. 운영 리스크 확인

- Render 백엔드 cold start로 첫 요청이 느릴 수 있다.
- 링크 공유 직전에 `/health`와 `/programs`를 한 번 열어 백엔드를 깨운다.
- Supabase 사용량 제한 경고가 있으면 DB/API 제한 가능성을 확인한다.
- Vercel, Render, Supabase 로그에서 500 오류가 반복되는지 확인한다.
- AI 요청 실패가 반복되면 Google API quota와 key 제한을 확인한다.
