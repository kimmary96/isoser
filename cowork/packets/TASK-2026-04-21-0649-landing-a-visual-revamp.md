## Goal

현재 랜딩 A는 카피 중심 텍스트 블록과 프로그램 카드 나열 위주라, 비개발자 방문자가 이소서의 실제 제품 흐름을 한눈에 이해하기 어렵다. 이번 작업은 랜딩 A를 "정보 허브 + 개인화 추천 + 지원 준비 플랫폼"이라는 정체성에 맞게 재배치하고, 핵심 기능을 실제 대시보드 화면 미리보기 카드로 보여주는 구조로 전환한다. 이력서 기능을 메인으로 올리지 않고, "추천 정확도와 지원 속도를 올리기 위해 이력 데이터가 필요한 서비스"라는 프레이밍을 유지한다.

대상 코드 범위는 `frontend/app/(landing)/landing-a/` 디렉터리 단일이며, placeholder 이미지 에셋이 필요한 경우에만 `frontend/public/landing-a/` 정적 에셋 디렉터리 추가를 허용한다. 랜딩 B, 대시보드, API, 백엔드는 변경하지 않는다.

## User Flow

1. 비로그인 방문자가 랜딩 A에 진입한다.
2. 히어로에서 "국비·지역 프로그램을 내 이력 기반으로 추천받는 서비스"라는 메인 가치를 즉시 인지한다.
3. 실시간 D-Day/마감 요약과 프로그램 탐색 진입 CTA를 본다.
4. 스크롤을 내리며 (a) 기존에 겪던 문제 → 이소서 해결 방식 비교, (b) 탐색→이력 등록→맞춤 추천→지원 문서→성과 저장→재활용 순환 플로우, (c) 기능 맛보기 이미지 카드 4개, (d) 추천 정확도 설명, (e) 성과 증명 보드(KPI 뼈대)를 본다.
5. 프로그램 둘러보기 또는 내게 맞는 프로그램 추천받기 CTA로 이탈한다.

## UI Requirements

렌더링 섹션 순서는 다음 11개를 기준으로 한다(기존 컴포넌트 재사용이 가능한 경우 재사용을 우선한다).

1. 상단 티커/네브바: 기존 유지.
2. 히어로: 메인 카피/서브 카피/CTA 교체. 메인 카피는 "흩어진 국비·지역 프로그램을 내 이력 기반으로 추천받고 바로 지원하세요" 계열. 서브 카피는 "이력과 활동을 등록하면 나에게 맞는 프로그램을 추천하고, 지원에 필요한 이력서·포트폴리오까지 바로 준비합니다." 계열. 주 CTA "내게 맞는 프로그램 추천받기", 보조 CTA "프로그램 둘러보기". 히어로 카피와 CTA 톤은 랜딩 B와 구분되는 형태로 유지한다.
3. 실시간 D-Day/마감 요약 스트립: 기존 `LandingATickerBar`와 히어로 live board 중 이미 렌더링되는 마감 요약 UI를 재사용해도 된다. 별도 신규 API나 신규 데이터 fetching은 추가하지 않는다. 기존 UI가 히어로 내부에 있을 경우 섹션 순서상 히어로 직후의 마감 요약 역할을 한다고 간주한다.
4. 검색/칩 필터바: 기존 유지.
5. 프로그램 카드 그리드(주인공): 기존 유지.
6. 차별점 비교 섹션: 좌측 "겪고 있는 문제"(여기저기 흩어진 프로그램 정보, 내 상황과 안 맞는 추천, 지원 때마다 문서 다시 작성), 우측 "이소서 해결 방식"(한곳에 모음, 이력 기반 개인화, 같은 흐름에서 지원 문서 생성).
7. 순환 플로우 섹션: 6단계 플로우(프로그램 탐색 → 이력/활동 등록 → 맞춤 추천 → 지원 문서 생성 → 참여 성과 저장 → 다음 추천/취업 준비 재사용)를 아이콘/단계 카드로 시각화. 한 줄 설명 외 장문 해설 금지.
8. 기능 맛보기 이미지 미리보기 섹션(신규 핵심): 실제 carousel 동작을 새로 만들지 않고, 데스크톱 2열/모바일 1열의 반응형 카드 그리드로 구현한다. 카드 순서는 (1) 프로그램 추천 캘린더, (2) 성과저장소 STAR 코치, (3) 이력서/포트폴리오 PDF, (4) 공고 매칭 점수. 각 카드에 제목, 한 줄 설명, 대시보드 스크린샷 영역을 배치한다. 이미지는 실제 대시보드 캡처가 있으면 사용하되, 미개발/미캡처 영역은 `frontend/public/landing-a/` 아래 placeholder 이미지로 먼저 자리 확보하고 같은 파일명 교체만으로 대응 가능한 구조로 한다.
9. 추천 정확도 설명 섹션: 프로필/활동/관심 분야/지역·상황 네 요소가 추천에 반영되는 구조를 간결한 도식으로 표현.
10. 성과 증명 보드(KPI 뼈대): "누적 추천 프로그램 조회 수", "이력 기반 추천 생성 수", "프로그램 지원용 문서 생성 수", "프로그램 참여 후기 수", "프로그램 이후 취업/면접 후기 수" 항목만 뼈대로 노출. 실제 수치와 후기 슬롯은 이번 작업 범위 밖이며, 값은 비어있거나 "집계 준비 중" 수준 텍스트로 둔다. 실제 API 연동 없음.
11. 하단 CTA/푸터: 기존 유지 가능하면 유지. 하단 CTA와 푸터는 하나의 종료 섹션으로 간주한다.

문구 수정 5건(섹션 교체 후 잔존 문구 기준):

- "지금 탐색할 프로그램을 빠르게 고릅니다" → "지금 프로그램을 빠르게 탐색해 보세요"
- "흩어진 경력을 한 번 정리하면, 공고마다 다시 꺼내 쓸 수 있습니다." → "흩어진 경력 한 번에 정리하고, 원하는 공고에 맞춰 작성해 보세요"
- "이미 구현된 흐름을 첫 화면에서 바로 이해할 수 있어야 합니다." → "AI를 통해 이력서를 준비하고, 공고 매칭과 출력까지"
- "프로그램을 찾았다면, 이제 지원 준비를 같은 흐름으로 이어가면 됩니다" → "준비가 되었다면 이력서를 준비해 보세요."
- 아래 메타 해설성 작은 문구는 전부 삭제:
  - "비로그인 상태에서는 우선 공고를 탐색하고, 로그인 후에는 이 결과를 추천 캘린더와 문서 워크플로우로 이어 붙입니다."
  - "지금 제품의 설득 포인트는 미래 기능이 아니라, 이미 연결된 작업 흐름입니다."
  - "탐색만 가능한 제품처럼 보이면 로그인 동기가 약해집니다. 그래서 랜딩 본문에서부터 로그인 후 워크플로우를 명시적으로 보여줍니다."
  - "로그인 후에는 추천 프로그램 캘린더, 성과저장소, 이력서, 자기소개서, 매치 분석이 같은 제품 안에서 이어집니다."

## Acceptance Criteria

- `/landing-a` 진입 시 위 11개 섹션 구조가 반영되어 있다. 섹션 순서와 존재 여부가 실제 렌더링에서 확인 가능하다.
- 히어로 메인 카피/서브 카피/CTA 문구가 본 packet 기준으로 교체되어 있고, 랜딩 B와 동일하지 않다.
- 기능 맛보기 이미지 미리보기 섹션에 4개 카드가 지정된 순서(프로그램 추천 캘린더 → 성과저장소 STAR 코치 → 이력서/포트폴리오 PDF → 공고 매칭 점수)로 노출된다.
- 각 맛보기 카드에 이미지 슬롯(실제 캡처 또는 placeholder)이 존재하고, `frontend/public/landing-a/` 아래 같은 파일명 에셋 교체만으로 이미지를 바꿀 수 있는 구조다.
- 성과 증명 보드 섹션에 5개 KPI 항목이 뼈대로 노출되며, 실제 수치 대신 "집계 준비 중" 또는 빈 placeholder 값으로 표기된다. 백엔드 API 호출은 없다.
- 문구 수정 5건이 반영되어 있고, 삭제 대상 4개 해설성 문구가 렌더링 결과에 존재하지 않는다.
- 기존 검색/칩 필터, 프로그램 카드 그리드, 티커, 네브바, 푸터의 기능 동작은 변경되지 않는다.
- 로그인 상태 분기 동작, 데이터 fetching 경로, API 계약은 바뀌지 않는다.
- `npm --prefix frontend run build` 및 `npm --prefix frontend run lint`가 통과한다.
- 모바일 뷰포트(375px 기준)에서 섹션 스택이 깨지지 않고 가로 스크롤이 발생하지 않는다.

## Constraints

- 변경 범위는 코드 기준 `frontend/app/(landing)/landing-a/` 내부로 한정한다. placeholder 이미지 에셋이 필요한 경우 `frontend/public/landing-a/` 정적 파일 추가만 예외로 허용한다. `landing-b`, `dashboard`, `api`, `backend`는 건드리지 않는다.
- 기존 데이터 페칭(`listPrograms`, `getProgramCount`)과 chip/keyword 정규화 로직은 재사용한다. 새로운 API 엔드포인트 추가 금지.
- 이미지는 `frontend/public/landing-a/` 이하 정적 경로에 두고 Next/Image 또는 기존 패턴을 따른다. 외부 CDN 의존 추가 금지.
- Tailwind 기반 스타일 유지. 외부 UI 라이브러리 신규 도입 금지.
- `CLAUDE.md`, `AGENTS.md`, `docs/*.md` 등 기준 문서 수정 금지.
- 기능 동작 추가가 아닌 구조/표현 리뉴얼이므로, 새로운 상태 관리나 클라이언트 훅 도입은 최소화한다.

## Non-goals

- 실제 KPI 수치 집계 및 API 연동.
- 실제 후기/리뷰 데이터 연결 또는 관리 화면 구축.
- 기능 맛보기 카드의 실제 대시보드 스크린샷 제작·촬영(현재 개발된 범위만 사용 가능, 나머지는 placeholder).
- 랜딩 B 수정, A/B 테스트 분배 로직 변경.
- 새로운 라우트/페이지 추가.
- 대시보드, 인증, 온보딩 흐름 변경.
- 번역(i18n) 확장.

## Edge Cases

- `listPrograms` 응답이 비어 있거나 에러일 때: 프로그램 카드 그리드가 기존 fallback UI를 유지하고, 다른 섹션(비교/플로우/맛보기/KPI 뼈대)은 정상 렌더된다.
- 이미지 에셋 누락: placeholder 이미지 또는 스켈레톤이 보이며 레이아웃이 무너지지 않는다.
- 모바일 뷰: 2열 맛보기 카드 그리드는 1열로 자연스럽게 스택되고, 비교 섹션의 좌우 구성은 상하 스택으로 전환된다.
- 느린 네트워크: 상단 히어로/카피/CTA는 데이터 없이도 즉시 렌더된다(SSR 기준 유지).
- 접근성: 맛보기 이미지에 의미 있는 alt 텍스트를 제공한다. 주요 CTA는 키보드 포커스가 가능하다.
- 다크/라이트 테마: 기존 랜딩 A의 컬러 토큰만 사용하며 신규 테마 도입 없음.

## Resolved Planning Decisions

- 기능 맛보기 4개 카드 각각에 들어갈 placeholder 이미지는 실제 대시보드 캡처 미리보기 풍의 정적 placeholder를 기본값으로 한다.
- 순환 플로우 섹션은 데스크톱 가로 스텝 인디케이터, 모바일 세로 스택으로 구현한다.
- KPI 뼈대 값은 "집계 준비 중" 텍스트로 통일한다.
- 기능 맛보기 섹션에는 카드별 CTA를 두지 않고, 섹션 하단 또는 하단 CTA 영역의 기존 CTA로 통합한다.
- 추천 정확도 설명 섹션의 4요소 도식은 텍스트 카드 4개로 구현하고, 별도 다이어그램 라이브러리나 복잡한 관계도는 추가하지 않는다.
