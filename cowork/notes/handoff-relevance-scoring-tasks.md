## 2. 이전 세션 결정 사항 요약

### 2-1. 프로그램 목록 UI 개선 방향 v2 (확정)

이소서 vs 부트텐트 비교 분석을 거쳐 결정된 방향성입니다.

**차별화 포지셔닝 한 문장**
> 부트텐트는 "전체 부트캠프를 잘 보여주는 곳"이고, 이소서는 "내 이력서 기준으로 나한테 맞는 부트캠프를 딱 3개 집어주는 곳"이다.

**카드 액션 버튼 전면 제거**
- 기존: `상세 보기 / 비교에 추가 / 지원 링크` 3개 버튼
- 변경: 우측 상단 찜(별) 버튼만 유지
- 근거: 비교 기능은 별도 비교 페이지에서 찜한 프로그램 기준으로 처리. 지원 링크는 상세 페이지 경유
- 카드 전체를 클릭 영역으로 → 상세 페이지 진입

**빈 상태 문구 숨김 처리**
- "프로그램 소개가 아직 등록되지 않았습니다", "일정 추후 공지", "지역 정보 없음", "태그 정보 없음" 등은 UI에서 라인 자체를 숨김
- 불가피한 경우 출처 표기로 대체 ("HRD넷 업데이트 예정")
- 필수 필드(제목, 마감일, 출처) 누락된 프로그램은 목록 노출 제외

**정보 밀도 상향**
- 카드 레이아웃: `출처·지역` / `제목` / `기간·형태·비용·카테고리` / `D-day 뱃지`
- 부트텐트 수준의 정보량 확보

**필터 보강 (부트텐트 벤치마킹)**
현재 이소서 필터는 5종(카테고리, 온/오프라인, 지역, 비용, 참여 시간)으로 부트캠프 탐색의 실질 의사결정 축이 빠져 있음.

| 필터 | 옵션 예시 | 우선순위 |
|---|---|---|
| 선발 절차 | 선발 절차 없음 / 서류 / 면접 / 코테 | 1순위 |
| 채용 연계 | 인재추천 / 채용우대 / 인턴십 연계 | 1순위 |
| 운영 기관 | HRD넷 / 고용24 / K-디지털 / K-Startup | 2순위 |
| 추천 대상 | 청년 / 경력단절 / 재직자 / 구직자 | 2순위 |
| 우수훈련기관 | 뱃지 필터 | 3순위 (데이터 확보 후) |

**정렬 옵션 재정의**
| 정렬 | 비고 |
|---|---|
| 맞춤 추천순 | 로그인 유저 기본값 |
| 마감 임박순 | 비로그인 기본값 |
| 최신순 | |
| 인기순 | 북마크 수 기준 (3순위) |
| 모집 중만 보기 | 토글 |

**전체 레이아웃 최종 구조**
```
헤더: 검색 + 필터 (드롭다운 7종)
─────────────────────────────────────────
🎯 내 맞춤 추천 프로그램 (40점 이상만)
  ┌─ 87% ─┐ ┌─ 72% ─┐ ┌─ 65% ─┐
  │ 근거3줄│ │ 근거3줄│ │ 근거2줄│
  └────────┘ └────────┘ └────────┘
─────────────────────────────────────────
🔥 마감 임박 (D-7 이내)
  ┌────┐ ┌────┐ ┌────┐
─────────────────────────────────────────
전체 프로그램 (847개) [정렬: 마감 임박순 ▾]
  ┌────┐ ┌────┐ ┌────┐
  ┌────┐ ┌────┐ ┌────┐
  [페이지네이션]
```

**비로그인 유저 처리**
- 맞춤 추천 섹션 자체는 보여주되, 카드들을 블러 처리
- 오버레이: "로그인하고 내 맞춤 추천 받기 →"
- 부트텐트와의 차별화를 첫 진입 순간에 각인

### 2-2. 관련도 점수 시스템 설계 (확정)

**방향 3 채택**: 숫자 + 자연어 근거 함께 노출

**최종 가중치 (주소 필드 추가 후)**
| 요소 | 가중치 |
|---|---|
| 희망 직무 일치 | 30% |
| 보유 스킬 일치 | 25% |
| 경험 도메인 일치 | 15% |
| 거주 지역 일치 | 15% |
| 준비도 | 10% |
| 행동 신호 | 5% |

**임시 가중치 (주소 필드 추가 전)**
| 요소 | 가중치 |
|---|---|
| 희망 직무 일치 | 35% |
| 보유 스킬 일치 | 30% |
| 경험 도메인 일치 | 20% |
| 거주 지역 일치 | 0% (비활성) |
| 준비도 | 10% |
| 행동 신호 | 5% |

**임계값 기반 뱃지 및 노출**
| 점수 | 뱃지 | 노출 위치 |
|---|---|---|
| 80~100 | 딱 맞아요 | 맞춤 추천 상단 |

```json
  "relevance_reasons": [
    "내 희망 직무 '풀스택 개발자'와 일치",

```

### 2-3. 저장소 현황 확인 결과

**스킬 필드 저장 형태**
- `profiles.skills`: Postgres `text[]`, 프론트 `string[] | null`
- `activities.skills`: Postgres `text[]`
- `programs.skills`: Postgres `text[]`, 프론트 `string[] | string | null` (파싱 방어 코드 필요)

**주의점**
- `programs.skills`가 현재 운영 환경에서 대체로 비어있음 (collector normalizer가 채우지 않음)
- Fallback으로 `title/summary/description/compare_meta` 토큰화가 이미 동작 중 (`backend/routers/programs.py` 1335 라인 부근)
- 이 때문에 "보유 스킬 일치 25%"의 실효성이 낮음 → **별도 Task(가칭 Task 2.5)로 collector normalizer 개선 필요. 이번 인수인계 범위 외이며, 본 문서 Task 1~3 작성 완료 후 별도 논의 예정**
- Task 1~3 구현 시 `programs.skills` 비어있는 케이스를 Edge Case로 반드시 다룰 것

**주소 필드**
- 현재 `profiles`에 주소 관련 필드 없음
- 추가 예정이며 스키마 결정 필요 (단일 `address` vs `sido`/`sigungu` 분리)

## 3. 작성할 Task Packet 3개 — 의존성과 병렬성

### 의존성 도식
```
Task 1 (선행) ──┐
                ├─→ Task 2 (Task 1 응답 스키마 안정화 후)
Task 3 (병렬) ──┘
```

- Task 1과 Task 3은 **병렬 작업 가능** (의존성 없음)
- Task 2는 Task 1의 응답 스키마(score_breakdown, relevance_reasons)가 안정화된 후 진행

### Task 1 (선행, P1): 프로그램 카드 리디자인 + 관련도 근거 표시
- 프론트: 카드 액션 버튼 제거, 찜(별)만 남김, 빈 상태 문구 숨김, 카드 클릭 → 상세 진입, 관련도 뱃지+숫자+근거 3줄 UI
- 백엔드: `compare-relevance`와 `/recommend` 응답에 `relevance_reasons`, `score_breakdown`, `relevance_grade`, `relevance_badge` 추가
- 가중치: 임시 가중치 사용 (주소 필드 없는 상태)
- 근거 문구 생성: 템플릿 기반 (LLM 최소화)

### Task 2 (Task 1 후행, P2): 주소 필드 마이그레이션 + 지역 매칭 활성화
- DB: `profiles` 테이블에 주소 필드 추가 (마이그레이션)
- 프론트: 프로필 입력 폼에 주소 필드 UI 추가
- 백엔드: 주소 정규화 유틸 추가, 지역 매칭 점수 계산, 임시 가중치 → 최종 가중치 전환
- 근거 문구에 지역 매칭 근거 포함

### Task 3 (Task 1과 병렬, P2): 프로그램 목록 페이지 구조 개편
- 필터 보강 (선발 절차, 채용 연계, 운영 기관, 추천 대상)
- 정렬 옵션 추가 (맞춤 추천순, 인기순, 모집 중 토글)
- 섹션 분리 (맞춤 추천 / 마감 임박 / 전체)
- 비로그인 유저 블러 + CTA 처리

## 4. Task Packet 필수 형식

YAML frontmatter 필수 필드:
- `id`: TASK-YYYY-MM-DD-HHMM-short-slug
- `status`: proposed
- `type`: feature
- `title`
- `planned_at`: ISO 8601
- `planned_against_commit`: **현재 commit hash 필수**. Cowork에서 `git rev-parse HEAD`로 확인 후 채울 것

권장 필드:
- `priority`: P1 (Task 1), P2 (Task 2, Task 3)
- `planned_by`: Claude (planning session)
- `depends_on`: Task 2는 Task 1 의존, Task 3는 독립

본문 섹션:
- Goal
- User Flow
- UI Requirements
- Acceptance Criteria
- Constraints
- Non-goals
- Edge Cases
- Open Questions

## 5. Task 1 작성 지시사항

**파일명**
`cowork/packets/TASK-2026-04-23-HHMM-program-card-redesign-with-relevance.md`

**Goal**
프로그램 목록 카드의 액션 버튼을 제거하고 찜 버튼만 남긴다. 관련도 점수를 숫자 + 자연어 근거 3줄로 노출한다. `compare-relevance`와 `/recommend` 응답에 근거 문구 생성에 필요한 필드를 추가한다. 빈 상태 문구는 숨기거나 출처로 대체한다.

**User Flow**
1. 유저가 `/programs` 진입
2. 맞춤 추천 섹션에서 관련도 80+ 카드 상단 노출, 근거 3줄 함께 노출
3. 관련도 40 미만은 맞춤 추천 섹션에서 제외
4. 카드 우측 상단 별 클릭 → 찜 토글
5. 카드 본문 영역 클릭 → 프로그램 상세 페이지 진입

**UI Requirements**
- 카드 우측 상단에 별 버튼만. 기존 `상세 보기 / 비교에 추가 / 지원 링크` 3개 버튼 제거
- 별 버튼은 이벤트 버블링 차단 (카드 클릭과 분리)
- 관련도 뱃지: 80+ `딱 맞아요`, 60-79 `추천`, 40-59 `조건 일치`
- 근거 문구 최대 3개, 기여도 8점 이상 요소만, 기여도 내림차순
- 빈 데이터는 라인 자체 숨김, "일정 추후 공지" 같은 문구 노출 금지

**근거 문구 템플릿 예시 (관련도 가중치 v2 기준)**
- 희망 직무: "내 희망 직무 '{target_job}'와 일치"
- 스킬: "{skill_list} {n}개 스킬 매칭"
- 경험: "{domain} 관련 활동 {n}건 보유" 또는 "내 프로젝트 '{activity_title}'와 같은 분야"
- 준비도: "자격증 {n}개 + 활동 {m}건으로 중급 과정에 적합" 또는 "첫 국비 과정으로 추천"
- 행동: "최근 북마크한 {category} 과정들과 유사"
- 지역: Task 2 후 활성화. "{sido} 거주 지역과 일치", "온라인 과정으로 지역 무관"

**Acceptance Criteria**
- 목록 카드에서 3개 액션 버튼이 더 이상 노출되지 않는다
- 카드 우측 상단 별 버튼으로 찜 토글이 가능하다
- 카드 본문 클릭 시 상세 페이지로 이동한다
- `compare-relevance` 응답에 `relevance_reasons`, `score_breakdown`, `relevance_grade`, `relevance_badge`가 포함된다
- 기존 응답 필드(`relevance_score`, `matched_skills` 등)는 유지된다
- `/recommend` 응답에도 동일 필드가 포함된다
- 관련도 40 미만 프로그램은 맞춤 추천 섹션에 노출되지 않는다
- 프로필/활동이 모두 비어있는 유저는 맞춤 추천 섹션 대신 CTA 안내가 노출된다
- 빈 상태 문구("지역 정보 없음" 등)가 UI에 노출되지 않는다
- 필수 필드(제목, 마감일, 출처) 누락된 프로그램은 카드로 노출되지 않는다

**Constraints**
- 이번 Task에서는 주소 필드 미도입, 임시 가중치 사용 (지역 매칭 0%)
- 근거 문구는 템플릿 기반으로 생성, LLM 호출 최소화
- 기존 응답 필드는 유지 (호환성)
- 카드 전체 클릭 영역화 시 별 버튼 영역은 이벤트 버블링 차단
- `programs.skills` 비어있는 운영 현실 고려, title/summary/description fallback 활용

**Non-goals**
- 주소 필드 마이그레이션 (Task 2 범위)
- 지역 매칭 점수 계산 (Task 2 범위)
- 필터 보강 (Task 3 범위)
- 정렬 옵션 추가 (Task 3 범위)
- 섹션 분리 (Task 3 범위)
- 비로그인 유저 블러 처리 (Task 3 범위)
- collector/normalizer의 skills 추출 로직 개선 (별도 Task)

**Edge Cases**
- 프로필·활동 모두 비어있는 유저 → 맞춤 추천 섹션에 CTA 안내로 대체
- 직무만 있고 스킬 없는 유저 → 직무 50% + 경험 30% + 지역 20% 재가중 (주소 추가 후 활성화)
- 활동만 있고 프로필 비어있는 유저 → `activities.skills`를 `profiles.skills`로 간주
- `programs.skills`가 비어있는 프로그램 → title/summary/description 토큰화 fallback
- 프론트에서 `programs.skills`가 `string | string[] | null` 세 형태로 올 수 있음 → 파싱 방어 코드
- 관련도 점수 동점인 경우 → 직무 > 스킬 > 경험 > 지역 > 준비도 > 행동 순으로 정렬

**Open Questions**

## 6. Task 2 작성 지시사항

**파일명**
`cowork/packets/TASK-2026-04-23-HHMM-address-field-and-region-matching.md`

**의존성**: Task 1 선행 필요 (응답 스키마 안정화 후)

**Goal**
`profiles`에 주소 필드를 추가하고 프로필 입력 폼에 입력 UI를 반영한다. 백엔드에 주소 정규화 유틸을 추가하고 지역 매칭 점수 계산을 구현한다. 관련도 가중치를 임시 → 최종으로 전환한다. 근거 문구에 지역 근거를 포함한다.

**User Flow**
1. 유저가 프로필 설정 페이지 진입
2. 주소 입력 필드(시/도 수준 선택 또는 자유 입력)를 채움
3. 저장 시 서버에서 시/도 정규화 값 산출 후 DB 저장
4. 이후 `/programs` 맞춤 추천에서 지역 매칭 근거가 함께 노출 ("서울 거주 지역과 일치")

**UI Requirements**
- 주소 입력 필드를 프로필 폼에 추가 (위치는 경력·학력 근처)

- 입력 형식은 자유 입력 또는 시/도 드롭다운 중 택일 (구현 시 결정)
- 구/동 단위는 매칭에 사용하지 않음을 안내 (선택적)

**Acceptance Criteria**
- `profiles` 테이블에 주소 필드가 추가된다 (신규 마이그레이션 파일)
- 기존 유저는 NULL 허용으로 기존 데이터 영향 없음
- 프로필 폼에서 주소 입력/수정/저장이 가능하다
- 백엔드 feature builder가 주소 원문을 시/도로 정규화한다
- 가중치가 임시 가중치(직무 35/스킬 30/경험 20/지역 0/준비도 10/행동 5)에서 최종 가중치(직무 30/스킬 25/경험 15/지역 15/준비도 10/행동 5)로 전환된다
- 지역 매칭 점수가 가중치 15%로 `score_breakdown.region`에 반영된다
- 주소 있는 유저는 근거 문구에 지역 매칭 항목 노출 가능
- 주소 없는 유저는 지역 가중치를 다른 요소에 재분배 (임시 가중치와 동일 동작)
- 시/도 완전 일치는 15점, 인접 시/도(수도권 등)는 10점
- 온라인 프로그램은 지역 무관 가점 12점 적용
- 혼합형(온·오프라인) 프로그램은 10점 적용
- 근거 문구에서 시/도 단위까지만 노출되고 구/동 단위는 노출되지 않는다

**Constraints**
- 주소는 민감 정보 → 근거 문구에서 시/도 단위까지만 노출, 구/동 단위 노출 금지
- 기존 마이그레이션 파일 수정 금지, 새 SQL 파일로 추가
- `@supabase/ssr` 쿠키 세션 유지
- Task 1이 선행 완료된 뒤 작업 (응답 스키마 안정화 후 지역 점수 추가)
- 주소 정규화 사전은 시/도 17개 기준 (특별시·광역시·도)

**Non-goals**
- 구/동 단위 매칭
- 지하철역·교통편 정보
- 주소 자동완성 API 연동 (차후 개선)
- 프로그램 측 지역 메타데이터 보강 (별도 Task)

**Edge Cases**
- 주소 미입력 유저 → 지역 가중치 재분배, Task 1의 임시 가중치 동작 유지
- 해외 주소 입력 → 매칭 불가로 처리, 근거 문구에서 지역 제외
- 프로그램 측 지역 데이터가 비어있는 경우 → 지역 매칭 0점, 근거 문구 제외
- 유저 주소가 도/시 혼합된 경우(세종시 등) → 특수 처리 필요
- 온라인 프로그램에 지역이 함께 표기된 경우 → 온라인 우선

**Open Questions**
- 주소 필드 스키마: 단일 `address` 문자열 vs `sido`/`sigungu` 분리 (권장: 원문 + 정규화 시/도 분리)

- 인접 시/도 정의 (수도권만 적용할지 광역권 전체 적용할지)

## 7. Task 3 작성 지시사항

**파일명**
`cowork/packets/TASK-2026-04-23-HHMM-programs-listing-page-restructure.md`

**의존성**: Task 1과 병렬 가능 (독립)

**Goal**
프로그램 목록 페이지의 전체 구조를 개편한다. 부트캠프 의사결정에 필요한 필터를 보강하고, 정렬 옵션을 추가하며, 페이지를 "맞춤 추천 / 마감 임박 / 전체" 세 섹션으로 분리한다. 비로그인 유저에게는 맞춤 추천 섹션을 블러 처리하고 로그인 CTA를 노출해 차별화 메시지를 첫 진입 순간 전달한다.

**User Flow**

비로그인 유저:
1. `/programs` 진입
2. 상단에 맞춤 추천 섹션이 블러 처리된 상태로 노출, "로그인하고 내 맞춤 추천 받기" CTA
3. 마감 임박 섹션과 전체 프로그램 섹션은 정상 노출
4. 필터/정렬 사용 가능
5. CTA 클릭 → 로그인 → 맞춤 추천 활성화

로그인 유저:
1. `/programs` 진입
2. 맞춤 추천 섹션에서 관련도 40+ 프로그램이 점수 내림차순 노출 (Task 1 결과 활용)
3. 마감 임박 섹션 (D-7 이내)
4. 전체 프로그램 섹션 (정렬 옵션: 맞춤 추천순/마감 임박순/최신순/인기순)
5. 모집 중만 보기 토글로 전체 ↔ 모집중 전환
6. 필터로 선발 절차/채용 연계/운영 기관/추천 대상 등 좁히기

**UI Requirements**

필터 영역:
- 기존 5종(카테고리, 온/오프라인, 지역, 비용, 참여 시간)에 추가
- 1순위 추가: 선발 절차, 채용 연계
- 2순위 추가: 운영 기관, 추천 대상
- 3순위 추가: 우수훈련기관 뱃지 필터 (데이터 확보 후)
- 드롭다운 형태 (부트텐트 패턴 참고)

정렬 영역:
- 정렬 옵션: 맞춤 추천순(로그인 시 기본), 마감 임박순(비로그인 기본), 최신순, 인기순(3순위)
- 모집 중만 보기 토글 (전체 결과 수와 모집중 결과 수 함께 노출)

섹션 분리:
- 섹션 1: 맞춤 추천 (로그인 유저, 관련도 40+ / 비로그인 유저는 블러)
- 섹션 2: 마감 임박 (D-7 이내, 모든 유저)
- 섹션 3: 전체 프로그램 + 페이지네이션
- 각 섹션 헤더에 카드 수 노출

비로그인 처리:
- 맞춤 추천 섹션: 블러된 카드 3장 (실제 데이터 또는 더미)
- 오버레이: "로그인하고 내 맞춤 추천 받기 →" CTA
- 클릭 시 로그인 페이지로 이동

**Acceptance Criteria**
- 필터에 선발 절차, 채용 연계 옵션이 추가된다
- 필터에 운영 기관, 추천 대상 옵션이 추가된다
- 필터 변경 시 결과 카운트가 즉시 업데이트된다
- 정렬 옵션에 맞춤 추천순(로그인 유저), 마감 임박순, 최신순이 포함된다
- 정렬 변경 시 카드 순서가 즉시 재정렬된다
- 모집 중만 보기 토글이 동작한다
- 페이지가 맞춤 추천 / 마감 임박 / 전체 세 섹션으로 분리된다
- 각 섹션 헤더에 카드 수가 표시된다
- 비로그인 유저는 맞춤 추천 섹션이 블러 처리되어 CTA와 함께 노출된다
- 비로그인 유저가 CTA 클릭 시 로그인 페이지로 이동한다
- 로그인 후 맞춤 추천 섹션이 정상 노출된다

**Constraints**
- Task 1의 카드 디자인을 그대로 사용 (병렬 작업이므로 카드 컴포넌트는 공유)
- 필터/정렬 변경은 URL 쿼리 파라미터에 반영 (공유 가능, 뒤로 가기 동작)
- 기존 페이지네이션 동작은 유지
- 비로그인 블러 처리 시 실제 추천 데이터를 노출하지 않음 (개인정보 누출 방지) — 더미 또는 인기 프로그램으로 대체
- 인기순 정렬은 북마크 집계 인프라 확보 후 활성화 (Phase 2)
- 필수 필드(제목, 마감일, 출처) 누락된 프로그램은 모든 섹션에서 노출 제외 (Task 1과 동일 규칙)

**Non-goals**
- 카드 디자인 변경 (Task 1 범위)
- 관련도 점수 계산 로직 (Task 1 범위)
- 백엔드 응답 스키마 변경 (Task 1 범위)
- 주소 필드 / 지역 매칭 (Task 2 범위)
- 우수훈련기관 데이터 수집 인프라 (별도 Task)
- 필터 결과의 무한 스크롤 전환

**Edge Cases**
- 필터 결과가 0개 → "조건에 맞는 프로그램이 없습니다" 빈 상태 + 필터 초기화 CTA
- 맞춤 추천 결과가 0개 (로그인 유저, 모두 40점 미만) → "프로필을 더 채우면 맞춤 추천이 향상돼요" CTA
- 마감 임박 결과가 0개 → 섹션 자체 숨김 (빈 섹션 노출 금지)
- 정렬 변경 시 페이지 1로 리셋
- 비로그인 유저가 필터 사용 중 로그인 → 필터 상태 유지
- URL 쿼리 파라미터 손상 시 → 기본값으로 fallback
- 모바일 뷰 → 필터를 가로 스크롤 칩 형태 또는 바텀시트로 (반응형 처리)

**Open Questions**
- 비로그인 블러 처리에 사용할 더미 카드 vs 인기 프로그램 카드 (권장: 인기 카드, 실제 데이터지만 개인화 아닌 것)
- 필터 옵션 데이터 소스 (선발 절차/채용 연계 등이 `programs` 테이블 어디에 저장되는지 확인 필요)
- 운영 기관 필터의 옵션 목록 (HRD넷/고용24/K-디지털/K-Startup 4종 외 추가 여부)
- 모바일 반응형 필터 UX (가로 스크롤 칩 vs 바텀시트)
- 추천 대상 필터 데이터가 `programs`에 있는지 확인 필요
